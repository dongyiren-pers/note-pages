# 第4章 分布式锁

在应用开发中，特别是 web 工程开发，通常都是并发编程，不是多进程就是多线程。这种场景下极易出现线程并发性安全问题，此时不得不使用锁来解决问题。在多线程高并发场景下，为了保证资源的线程安全问题，jdk 为我们提供了 synchronized 关键字和 ReentrantLock 可重入锁，但是它们只能保证一个工程内的线程安全。在分布式集群、微服务、云原生横行的当下，如何保证不同进程、不同服务、不同机器的线程安全问题，jdk 并没有给我们提供既有的解决方案。此时，我们就必须借助于相关技术手动实现了。目前主流的实现有以下方式：

  • 基于 mysql 关系型实现
  • 基于 Redis 非关系型数据实现
  • 基于 zookeeper/etcd 实现

本章将会全面深入、全程手撸代码式的讲解这三种分布式锁的实现。并深入源码讲解第三方分布式锁框架。

# 4.1 传统锁

# JVM 本地锁

基于 syncronized 加锁：

public synchronized void deduct() {
    Stock stock = stockMapper.selectOne(new QueryWrapper<stock>().eq("product_code", "1001"));
    if (stock != null & stock.getCount() > 0){
        stock.setCount(stock.getCount() - 1);
        stockMapper.updateById(stock);   
    }
}

基于 ReentrantLock 加锁：

private ReentrantLock lock = new ReentrantLock();

public void deduct() {
    lock.lock();
    try {
        Stock stock = stockMapper.selectOne(new QueryWrapper<stock>().eq("product_code", "1001"));
        if (stock != null & stock.getCount() > 0){
            stock.setCount(stock.getCount() - 1);
            stockMapper.updateById(stock);   
        }
    } finally {
        lock.unlock();
    }
}

JVM 本地锁机制失效 的场景：

  • 多例模式：多例模式下，对象锁无法锁住多个实例。
  • 与事务共存：RC 隔离级别下，线程 A 释放本地锁但尚未提交事务时，线程 B 加锁成功后，读到的是线程 A 更新前的数据。
  • 集群部署：JVM 本地锁显然无效。

# MySQL 锁

简单的业务场景可以通过一个 SQL 语句解决超卖问题：

UPDATE db_stock SET count = count - 1 WHERE product_code = '1001' AND count >= 1;

事实上是基于 MySQL 悲观锁 实现的，UPDATE、INSERT、DELETE 这些 DML 语句会自动加上 X 锁。

悲观锁

基于悲观锁的示例：

@Transactional
public void deduct() {
    Stock stock = stockMapper.queryByCode("1001");
    if (stock != null & stock.getCount() > 0){
        stock.setCount(stock.getCount() - 1);
        stockMapper.updateById(stock);   
    }
}

public interface StockMapper extends BaseMapper<Stock> {
    @Select("SELECT * FROM db_stock WHERE product_code = #{productCode} FOR UPDATE")
    Stock queryByCode(@Param("productCode") String productCode);
}

注意锁的级别，行级锁是基于索引的，如果一条 SQL 语句不走索引，会使用临键锁把整张表锁住。

悲观锁的缺点：

  • 性能较低：悲观锁导致吞吐量降低。
  • 死锁问题：对多条数据加锁时，加锁顺序不一致可能导致死锁问题。
乐观锁

在表中新增一列：时间戳 timestamp 或版本号 version。

基于乐观锁的示例：

public void deduct() {
    Stock stock = stockMapper.selectOne(new QueryWrapper<stock>().eq("product_code", "1001"));
    if (stock != null & stock.getCount() > 0){
        stock.setCount(stock.getCount() - 1);
        stock.setVersion(stock.getVersion() + 1);
        int rows = stockMapper.update(stock, new QueryWrapper<stock>().eq("id", stock.getId())
                           				.eq("version", stock.getVersion()));
        if (rows == 0) {
            try {
                Thread.sleep(20);		// 防止栈溢出
            } catch (InterruptedException e) {
                e.printstackTrace();
            }
            deduct();
        }
    }
}

乐观锁的缺点：

  • 性能问题：高并发情况下，重试可能性很大，性能极低。此外，DB 读写分离情况下，重试可能性极大，性能就更低了。
  • ABA 问题：可通过时间戳或版本号解决。

# Redis 乐观锁

Redis 的乐观锁是基于 WATCH 命令和事务实现的：

> WATCH stock  			# 监听 Key 的值
OK
> GET stock
"5000"
> MULTI					# 开启事务
OK
> SET stock "4999"		# 执行此命令前已被其他请求修改
QUEUED
> EXEC					# 提交事务
(nil)					# 修改失败
