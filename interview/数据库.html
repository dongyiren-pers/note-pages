# MySQL

# MySQL 的存储引擎

阿里健康

InnoDB：InnoDB 是 MySQL 的默认存储引擎，也是最常用的存储引擎之一。它 支持事务处理、行级锁和外键约束 等高级功能。InnoDB 提供了较好的并发性能和数据完整性，适用于大多数应用场景。

MyISAM：MyISAM 是 MySQL 早期版本中的一种常用存储引擎。它对于读密集型应用具有较好的性能，但 不支持事务处理、行级锁和外键约束。MyISAM 适用于只读或很少更新的应用，例如博客等。

MEMORY ：该存储引擎将数据存储在内存中，因此读写速度非常快。它适用于临时表、缓存数据或其他需要快速访问的数据。然而，由于数据存储在内存中，一旦数据库重启，数据将丢失。

# InnoDB 与 MyISAM 的区别

字节
  • InnoDB 支持事务处理、行级锁和外键约束等高级功能，而 MyISAM 不支持。
  • InnoDB 是聚集索引，MyISAM 是非聚集索引。InnoDB 的 B+ 树主键索引的叶子节点就是数据文件，辅助索引的叶子节点是主键的值；而 MyISAM 的 B+ 树主键索引和辅助索引的叶子节点都是数据文件的地址指针。
  • InnoDB 不保存表的总行数，因此执行 SELECT COUNT(*) FROM table 时需要对整个表进行全表扫描，而 MyISAM 通过一个变量保存总行数，因此执行这条语句时速度更快。InnoDB 不保存表的具体行数的原因是因为它的事务特性。在同一时刻，对于不同的事务而言，表中的行数可能是不一样的。

# SQL 的执行流程

腾讯PCG

SQL 查询语句的执行流程：连接器 → 分析器 → 优化器 → 执行器 → 存储引擎

SQL 更新语句的执行流程：与 SQL 查询语句相似，此外会涉及到写 binlog、redolog 和 undolog 日志。

# 索引的分类

快手本地生活

常见的索引分类如下：

  • 根据字段特性划分：普通索引、唯一索引、主键索引、全文索引
  • 根据物理实现划分：聚簇索引、非聚簇索引
  • 根据数据结构划分：B+ 树索引、B 树索引、Hash 索引、Full-text 索引
  • 根据字段个数划分：单列索引、联合索引（又称复合索引、组合索引）

# 为什么 InnoDB 选择 B+ 树作为索引

快手商业化
  • 与非平衡二叉树相比：数据量过大时，非平衡二叉树的 高度很高，磁盘 IO 次数过多。此外，极端情况下二叉树可能 退化成链表，此时查找效率为 O(n)。
  • 与红黑树相比：红黑树通过自旋解决了二叉树退化成链表的问题，但在数据量过大的情况下，红黑树的 高度仍然很高，磁盘 IO 次数仍然过多。
  • 与 B 树索引相比：B+ 树 非叶子节点不存储数据，只存储索引，可以存储更多的索引。当 B+ 树为三层的情况下，最多可以存放 1170 * 1170 * 16 = 21,902,400 条数据（16KB / (8B + 6B) = 1170 个）。即 千万级别 的表，只要经过 3 次磁盘 IO 就可以拿到想要的数据了。
  • 与哈希索引相比：如果是等值查询，那么哈希索引明显有绝对优势，因为只需要经过⼀次算法即可找到相应的键值。如果是 范围查询检索，这时候 哈希索引就 毫无用武之地 了，因为原先是有序的键值，经过哈希算法后，有可能变成不连续的了，就没办法再利用索引完成范围查询检索。

# 什么是覆盖索引

从非主键索引中就能查到的记录，而不需要查询主键索引中的记录，避免了回表的产生，减少了树的搜索次数，显著提升性能。详见 覆盖索引。

# 什么是索引下推

当执行一个包含 索引列条件 的查询时，MySQL 5.6 可以在存储引擎层级执行部分查询条件，从而减少从存储引擎返回到 Server 层的数据，进而减少回表次数。详见 索引下推。

# 什么是索引合并

美团

从 MySQL 5.1 开始，引入了 index merge 优化技术，对同一个表可以使用多个索引分别进行条件扫描，然后将结果进行合并处理。详见 索引合并。

合并方式分为三种：intersection（取交集）、union（取并集）、Sort-Union（取有序并集）。

# 索引失效的场景

  • 不满足最左前缀原则：最左边的字段没有出现在查询条件中。
  • SELECT * 的范围查询：对 非聚簇索引列进行范围查询且结果列不满足覆盖索引条件 时，有极大概率不走索引。
  • 模糊匹配占位符位于条件首部：模糊查询时（LIKE 语句），模糊匹配占位符位于条件首部。
  • 索引列参与运算：索引列 参与了运算，会导致全表扫描，索引失效。
  • 索引列作为函数参数：索引列 参与了函数处理，会导致全表扫描，索引失效。
  • 索引列发生类型隐式转换：参数类型与索引列类型不匹配，导致索引列发生类型隐式转换，索引失效。
  • 使用 OR 关键字：查询条件使用 OR 关键字，而 其中一个字段没有创建索引，则会导致整个查询语句索引失效。
  • 对两列进行对比操作：如果两个列数据都有索引，但在查询条件中 对两列数据进行了对比操作，则会导致索引失效。
  • NOT EXISTS：查询条件使用 NOT EXISTS 时，索引失效。
  • 使用 ORDER BY：使用 ORDER BY 且结果列不满足最左前缀原则 时，索引失效。

# 如何通过 EXPLAIN 分析执行计划

阿里本地生活腾讯PCG

通过 EXPLAIN 分析执行计划：

  • type：数据访问类型
    • const：使用主键或唯一索引进行等值查询
    • ref：使用非唯一索引等值查询
    • index_merge：使用了索引合并的优化逻辑
    • range：索引范围查询
    • index：索引全表扫描
    • all：全表扫描
  • possible_keys：可能用到的索引
  • key：实际用到的索引
  • rows：预计扫描的行数
  • Extra：附加信息
    • Using where：使用了 WHERE 条件
    • Using index：使用了覆盖索引
    • Using index condition：使用了索引下推

只列举了常考字段，详见 EXPLAIN 执行计划。

# 事务的四个特性

阿里飞猪
  • 原子性（Atomicity）：指⼀个事务中的操作要么全部成功，要么全部失败
  • 一致性（Consistency）：指事务将数据库从一个一致性状态变到另一个一致性状态。
  • 隔离性（Isolation）：指的是⼀个事务的修改在最终提交前，对其他事务是不可⻅的。
  • 持久性（Durability）：指的是⼀旦事务提交，所做的修改就会永久保存到数据库中。

# 如何保证事务的 ACID 特性

阿里健康
  • 原子性 由 undo log 保证，它记录了需要回滚的⽇志信息，事务回滚时撤销已经执⾏成功的 SQL。
  • ⼀致性 由其他三⼤特性保证，程序代码要保证业务上的⼀致性 。
  • 隔离性 由 MVCC 和 锁 来保证。
  • 持久性 由内存和 redo log 来保证，MySQL 修改数据同时在内存和 redo log 记录这次操作，宕机时从 redo log 恢复。

# 事务隔离级别

阿里
  • 读未提交（READ-UNCOMMITTED）：最低的隔离级别，允许读取尚未提交的数据变更，这可能会导致脏读、不可重复读和幻读。
  • 读已提交（READ-COMMITTED, RC）：允许读取并发事务已经提交的数据，可以阻止脏读，但是不可重复读和幻读仍有可能发生。
  • 可重复读（REPEATABLE-READ, RR）：对同一字段的多次读取结果是一致的，除非数据被本身事务自己修改，可以阻止脏读和不可重复读，但幻读仍有可能发生。
  • 可串行化（SERIALIZABLE）：最高的隔离级别，完全服从 ACID 的隔离级别。事务依次逐个执行，这样事务之间就完全不可能产生干扰。该级别可以防止脏读、不可重复读以及幻读。

# 读已提交和可重复读是如何实现的

快手
  • 读已提交隔离级别下，每个快照读都会生成并获取最新的 Read View。
  • 可重复读隔离级别下，只有在同一个事务的第一个快照读才会创建 Read View，之后的每次快照读都使用的同一个 Read View。

# MySQL 的锁有哪些

滴滴京东腾讯

根据加锁的范围，可以分为 全局锁、表级锁和行锁 三类。

  • 表级锁有：表锁、元数据锁、意向锁、AUTO-INC 锁。
  • 行级锁有：记录锁、间隙锁、临键锁。

# bin log、redo log 和 undo log 的功能

字节

binlog 的功能：

  • 主从复制 ：在 Master 端开启 binlog ，然后将 binlog 发送到各个 Slave 端， Slave 端重放 binlog 来达到主从数据一致。
  • 数据恢复 ：通过使用 mysqlbinlog 工具来恢复数据。

redo log 的功能：

  • 实现事务的持久性，让 MySQL 有 crash-safe 的能力，能够保证 MySQL 在任何时间段突然崩溃，重启后之前已提交的记录都不会丢失；
  • 将写操作从 “随机写” 变成了 “顺序写”，提升 MySQL 写入磁盘的性能。

undo log 的功能：

  • 实现事务回滚，保障事务的原子性。事务处理过程中，如果出现错误或用户执行了 ROLLBACK 语句，MySQL 可以利用 undo log 中的历史数据将数据恢复到事务开始之前的状态。
  • 实现 MVCC（多版本并发控制）。MVCC 是通过 ReadView + undo log 实现的。undo log 为每条记录保存多份历史数据，MySQL 在执行快照读（普通 SELECT 语句）的时候，会根据事务的 Read View 里的信息，顺着 undo log 的版本链找到满足其可见性的记录。

# MySQL 的主从复制

百度TPG

在 MySQL 的主从复制中，主库会将变更写入 binlog，从库通过连接到主库来获取并复制主库的 binlog。主从复制具体详细过程如下：

  • 主库在收到客户端提交事务的请求之后，会先写入 binlog，再提交事务，更新存储引擎中的数据，事务提交完成后，返回给客户端 “操作成功” 的响应。
  • 从库会创建一个专门的 I/O 线程，连接主库的 log dump 线程，来接收主库的 binlog 日志，再把 binlog 信息写入 relay log 中继日志 里，再返回给主库 “复制成功” 的响应。
  • 从库会创建一个用于回放 binlog 的线程，去读 relay log 中继日志，然后回放 binlog 更新存储引擎中的数据，最终实现主从的数据一致性。
